# 动态漫剧创作工具设计文档

- 日期：2026-03-25
- 阶段：第一阶段 / 子项目 A
- 文档目标：明确第一版产品范围、核心流程、模块边界和成功标准，为后续实施计划提供依据

## 1. 项目背景

目标是做一个面向小说、短篇作者的 Web 创作工具。用户输入单章节文本，系统自动生成一版可编辑的动态漫剧草稿，用户在站内完成中度编辑后，可以预览并导出 MP4。

第一版不追求完整平台能力，也不优先追求产品护城河。优先级是：

1. 先把从文本到成片的可用闭环跑通
2. 保证生成结果可编辑、可修正、可继续迭代
3. 尽量复用现成模型、第三方 API 和开源工具，降低首版复杂度

## 2. 目标用户

第一版主要服务以下用户：

- 已经有小说章节、短篇故事或剧情片段的作者
- 希望把文字内容快速转成适合传播的动态漫剧视频
- 不希望从零做视频剪辑，但又希望对分镜、台词、角色形象和场景有一定控制权

这一版不是面向普通娱乐用户，也不是面向专业后期创作者。

## 3. 第一版产品定位

第一版定位为 `Web 创作台`，而不是内容社区、发布平台或专业视频编辑器。

产品核心价值是：

- 将单章节文本自动拆解为可编辑的漫剧草稿
- 让作者在“故事结构”和“镜头内容”层面进行中度编辑
- 最终导出一个可发布的 MP4 视频

## 4. 第一版范围

### 4.1 In Scope

第一版明确包含以下能力：

- 输入单章节文本创建项目
- 自动生成结构化剧本稿
- 自动生成动态漫剧草稿
- 中度编辑
- 站内预览
- 导出 MP4
- 失败任务局部重试
- 项目资产保存与再次打开继续编辑

其中，中度编辑包括：

- 调整分镜顺序
- 调整镜头时长
- 修改旁白和对白
- 替换角色形象
- 替换场景图
- 调整字幕文案
- 调整配音音色
- 单独重生成某个分镜相关素材

### 4.2 Out of Scope

第一版明确不做以下能力：

- 多章节连续剧情生成
- 整本小说拆解和批量出片
- 高级时间轴编辑器
- 多轨道自由拖拽剪辑
- 复杂关键帧动画编辑
- 社区、发布、分发、互动能力
- 多人协作
- 商业化计费系统
- 自研基础模型训练平台
- 工业级强角色一致性

## 5. 创作结果定义

一次成功创作的完成态定义为：

1. 用户输入单章节文本并创建项目
2. 系统产出一版结构化剧本稿
3. 用户可对剧本稿做必要调整
4. 系统根据确认后的剧本稿生成动态漫剧草稿
5. 用户在创作台中进行中度编辑
6. 用户在站内预览成片效果
7. 用户发起导出并获得 MP4

这意味着第一版不是单纯“一键生成视频”，而是“自动生成 + 中度编辑 + 最终导出”的创作流程。

## 6. 视觉与成片方向

第一版的成片方向定义为：

- 风格：偏日漫 / 国漫分镜感
- 形态：动态漫剧，而不是纯绘本视频或纯口播视频
- 动效：以轻动画、镜头推进、转场和字幕节奏为主
- 一致性要求：主要角色尽量保持一致，但允许局部轻微漂移

第一版不追求复杂角色骨骼动画或高强度动作演出，更适合用“静态图 + 镜头运动 + 轻动态效果 + 配音字幕”的方式完成动态漫剧表达。

## 7. 产品流程

### 7.1 核心用户流

推荐将一次创作流程收敛为以下步骤：

1. 项目创建
2. 文本输入
3. 剧本拆解
4. 剧本确认
5. 素材生成
6. 草稿预览
7. 分镜编辑
8. 站内整片预览
9. 导出 MP4

### 7.2 关键设计原则

- 先结构化，再生成素材
- 先确认剧本稿，再消耗生成成本
- 分阶段保存中间产物
- 支持局部替换和局部重试
- 以“编辑故事和分镜”为主，而不是“手搓专业视频轨道”

## 8. 创作台页面结构

第一版 Web 创作台建议拆成 5 个主要页面或功能区。

### 8.1 项目创建页

负责收集项目基础输入：

- 章节文本
- 题材模板
- 画风模板
- 目标时长
- 配音风格

### 8.2 剧本拆解确认页

展示自动生成的结构化剧本稿，主要包含：

- 角色卡
- 场景卡
- 剧情摘要
- 分镜列表
- 旁白和对白草稿

这一页允许用户：

- 修改角色名
- 删除或合并分镜
- 调整旁白和对白
- 对剧情节奏做轻量修正

这是第一版的关键页，因为它决定后续生成是否具备可控性。

### 8.3 素材生成页

展示各阶段生成状态，例如：

- 角色图生成中
- 场景图生成中
- 分镜图生成中
- 配音生成中

要求支持失败项单独重试，避免整项目重复生成。

### 8.4 分镜创作台

这是第一版的主编辑区域，建议布局为：

- 左侧：分镜列表
- 中间：当前分镜预览
- 右侧：属性编辑区
- 下方或弹层：整片预览入口

该页面应支持本次定义的中度编辑能力。

### 8.5 导出页

负责：

- 展示导出配置
- 发起导出任务
- 呈现导出进度
- 提供 MP4 下载入口

## 9. 核心数据流

建议将每次创作抽象为一个 `Project`，并围绕项目保存核心资产。

### 9.1 核心资产

- 原始文本
- 剧本结构稿
- 角色设定
- 场景设定
- 分镜列表
- 视觉资产
- 音频资产
- 时间轴草稿
- 导出产物

### 9.2 推荐数据流

1. 用户提交章节文本和基础参数
2. 内容理解能力产出结构化剧本稿
3. 用户确认或修改剧本稿
4. 编排层根据剧本稿批量生成素材
5. 系统合成第一版动态漫剧草稿
6. 用户在创作台里继续调整
7. 用户触发导出并获得 MP4

这个数据流的核心目标是可控和可恢复，而不是单次黑盒生成效果最大化。

## 10. 系统架构

第一阶段建议使用“前后端产品层自建 + 生成能力外接 + 导出链路自控”的方式。

### 10.1 模块划分

#### A. Web 前端

负责：

- 项目创建
- 剧本确认
- 分镜编辑
- 整片预览
- 导出状态展示

#### B. 应用后端

负责：

- 项目管理
- 任务调度
- 状态流转
- 资产记录
- 导出任务发起

这一层是业务中枢，但第一版只需要轻量编排，不需要过深的平台化设计。

#### C. 内容理解能力

负责将章节文本转换为结构化剧本稿，至少产出：

- 角色列表
- 场景列表
- 分镜列表
- 旁白
- 对白
- 每镜头的情绪、镜头建议和时长建议

第一版优先复用现成 LLM，不优先自建复杂文本理解引擎。

#### D. 素材生成编排层

负责根据结构化剧本稿调用外部能力，生成：

- 角色图
- 场景图
- 分镜图
- 配音
- 背景音乐和音效
- 轻动画相关素材或参数

这一层需要保持“可替换但不过度抽象”的原则。只做够用的 provider 适配，不提前做复杂平台。

#### E. 渲染导出服务

负责：

- 站内预览版本拼装
- 字幕、图片、音频和时长的合成
- MP4 导出

这一层建议自己掌控，因为它直接关系到“编辑后导出”的产品闭环。

#### F. 资产存储层

负责保存：

- 项目 JSON
- 图片资源
- 音频资源
- 视频资源
- 导出记录

其目标是保证项目可以继续编辑，而不是一次生成后即丢弃。

## 11. 技术策略

第一版的技术策略明确如下：

- 优先快速迭代，而不是优先构建护城河
- 能借开源工具就借开源工具
- 能接第三方模型就接第三方模型
- 产品侧只自建最必要的业务流程和创作体验

### 11.1 优先自建的部分

- 前端创作台
- 后端项目流转与状态管理
- 分镜编辑逻辑
- 预览拼装与导出流程
- 项目资产保存

### 11.2 优先复用的部分

- LLM 文本理解和润色
- 文生图和角色图生成
- TTS 配音
- 背景音乐和音效
- 图像轻动画或图转视频能力
- FFmpeg 等导出相关工具链

### 11.3 第一版不建议做深的部分

- 高度定制的模型训练体系
- 复杂模型编排平台
- 过度抽象的多供应商能力中心
- 围绕未来商业化的大量预埋架构

## 12. 错误处理与恢复策略

第一版应将“失败可恢复”作为核心能力。

### 12.1 错误处理原则

- 分阶段失败，不整项目报废
- 中间产物必须保留
- 每个失败项应有明确状态和原因
- 支持单个素材重试与替换
- 导出过程异步化

### 12.2 典型恢复方式

- 某个分镜出图失败：只重生成该分镜图
- 配音失败：只重试配音任务
- 某个角色形象不满意：单独替换角色图
- 某个场景效果差：单独换场景图
- 导出失败：在不改动项目资产的前提下重新导出

## 13. 风险与实现边界

第一版最容易出现的风险包括：

- 直接将整段文本丢给黑盒视频模型，导致结果不可控
- 过早追求强角色一致性，拖慢整体迭代
- 先做重型视频编辑器，导致产品复杂度失控
- 生成失败后只能整链路重跑，成本和体验都很差
- 没有项目资产层，导致结果不可回溯、不可继续编辑

对应的设计边界是：

- 优先保证结构化和可编辑性
- 动态效果先做轻量方案
- 重视项目资产保存
- 所有生成结果都应支持局部修正

## 14. 验证口径

第一版的验证建议分三层。

### 14.1 链路验证

验证是否能从单章节文本稳定走到“站内可预览草稿 + MP4 导出”。

### 14.2 产品验证

验证用户是否能实际完成以下行为：

- 修改分镜顺序
- 修改对白和旁白
- 替换角色图和场景图
- 调整镜头时长
- 重新生成局部素材

### 14.3 效果验证

重点关注：

- 剧情拆解是否基本合理
- 主要角色是否大体一致
- 成片节奏是否具有漫剧感
- 用户是否感到“可改、可控”

## 15. MVP 成功标准

第一版达到以下标准，即可认为 MVP 基本成立：

- 用户可以创建项目并输入单章节文本
- 系统可以生成一版结构化剧本稿
- 用户可以对剧本稿做轻量修正
- 系统可以生成动态漫剧草稿
- 用户可以进行中度编辑
- 用户可以在站内预览整片
- 用户可以导出 MP4
- 失败任务支持局部重试
- 项目关闭后可以再次打开并继续编辑

## 16. 第一版不承诺的结果

第一版不承诺以下结果：

- 动画工业级的角色一致性
- 多章节剧情连续性
- 专业视频软件级编辑自由度
- 所有题材和所有文本类型都稳定高质量生成
- 一键生成即可直接发布且无需人工调整

## 17. 推荐实施方向

第一阶段建议聚焦在以下子项目：

`文本 -> 结构化剧本稿 -> 动态漫剧草稿 -> 中度编辑 -> 站内预览 -> 导出 MP4`

这是当前最适合的切入点，因为它同时满足：

- 面向明确用户群体
- 有真实可用的创作闭环
- 范围可控
- 便于快速迭代
- 可以大量复用现成模型和开源能力

后续如果第一版验证成立，再继续考虑：

- 多章节项目
- 更多画风模板
- 更强角色一致性
- 更丰富的轻动画能力
- 更细的编辑能力
